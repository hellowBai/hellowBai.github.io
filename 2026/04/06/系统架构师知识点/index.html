系统架构师知识点

Allen
2026-04-06
系统架构, 软考高级

SOA（面向服务架构）

特点：

  • 无状态原则
  • 单一实例原则
  • 自包含模块原则
  • 服务松耦合
  • 粗颗粒度

ESB 是实现 SOA 架构的一种方式。

ESB 的功能：

  • 地址透明支持路由寻址
  • 支持各种协议
  • 支持服务注册命名
  • 支持各种消息格式发送
  • 支持各种数据解析

SOA 的几个技术点：

  • SOAP：使用 XML 作为一种通信协议，包含四个部分：
    1. SOAP 封装
    2. SOAP 编码规则（数据类型）
    3. RPC 调用
    4. SOAP 绑定
  • REST 接口：
    • GET、POST、PUT、DELETE、HEAD、PATCH
    • 五个原则：
      1. 所有操作都是无状态
      2. 所有操作不改资源标识
      3. 具有唯一的资源标识
      4. 所有事物都抽象成资源
      5. 使用通用接口访问资源
  • WSDL：
    • 基本属性：服务器在哪、做些什么、如何访问
  • UDDI：
    • 包含服务描述与发现

模型

几种模型：

  • 原型模型
  • 瀑布模型
  • 螺旋模型
  • RUP 模型（统一过程）

螺旋模型

每个阶段包含四个步骤：

  • 制定计划
  • 风险分析
  • 实施工程
  • 客户评估

RUP（统一过程）

四个阶段：

  • 构思阶段 I
  • 细化阶段 E
  • 构造阶段 C
  • 移交阶段 T

九个核心工作流：

  • 六大技术流：业务建模、需求、分析与设计、实现、测试、部署
  • 三大支持流：配置变更、环境、项目管理

原型模型

类型：

  • 水平（页面）
  • 垂直（复杂算法）
  • 抛弃
  • 演化型

瀑布模型

阶段：

  • 需求分析
  • 软件设计
  • 系统设计
  • 软件实现
  • 单元测试
  • 集成测试
  • 系统测试
  • 运行与维护

ABSD（基于架构的软件设计）

六个步骤：

  1. 需求设计
  2. 架构设计
  3. 架构文档化
  4. 架构复审核
  5. 架构开发实现
  6. 架构演化

需求设计

  • 需求获取
  • 需求抽象成类
  • 类打包成组
  • 类打包成构件
  • 需求审核

架构设计

  • 架构分析
  • 架构映射组件
  • 分析构件之间的关系
  • 获取架构
  • 构件审核

架构文档化

  • 获取架构需求的质量说明书
  • 获取架构的规格说明文档

架构实现

  • 架构分析与设计
  • 构件的开发
  • 构件的组合
  • 测试
  • 构件的需求评审

系统架构模型方法

面向对象建模

得到的模型是对象模型。

信息工程建模

得到的模型是实体-联系图。

结构化建模

得到的模型是数据流图（DFD）。

软件生命周期（SDLC）

  • 可行性分析
  • 需求分析
  • 系统设计（概要设计、详细设计）
  • 实现/编码
  • 测试阶段（单元测试、集成测试、系统测试、验收测试）
  • 软件部署和维护

软件系统的质量属性

  • 开发阶段关注：易扩展性、可重用性、易理解性、可测试性、可维护性、可移植性
  • 运行阶段关注：安全性、可用性、性能、可靠性、鲁棒性、可伸缩性、互操作性

其他概念：

  • 敏感点：系统中某个特性，改动会影响多个质量属性
  • 权衡点：影响多个质量属性的改动
  • 风险点：系统中存在的隐患
  • 非风险点：某个改动不会带来隐患，往往以“XXX 可以被接受、可以被改变”结尾

系统测试步骤

  1. 制定计划
  2. 设计用例
  3. 执行系统测试
  4. 缺陷管理与改正

软件五大风格

  1. 调用/返回风格：分层架构、主程序和子程序、面向对象
  2. 虚拟机风格：解释器风格、规则系统
  3. 数据流风格：管道过滤、批处理系统
  4. 独立构件风格：进程通信、事件驱动
  5. 以数据为中心风格：仓库系统、黑板系统、超文本系统

MVC

  • M：一般在数据库存取数据
  • C：控制处理数据，通常处理用户输入，从 View 获取数据，向模型输送数据
  • V：显示视图，一般根据数据来显示

J2EE

  • Model：entity bean、session bean
  • View：JSP
  • Control：Servlet

特点：

  • 分层开发
  • 组件化开发
  • 容器管理
  • 标准化接口

云原生架构设计

七个原则：

  1. 服务化原则
  2. 弹性原则
  3. 可观测原则
  4. 韧性原则
  5. 零信任原则
  6. 可演变原则
  7. 自动化原则

边云协同分类：

  • 资源协同
  • 数据协同
  • 智能协同
  • 业务管理协同
  • 应用管理协同
  • 服务协同

架构评估方法

SAAM 和 ATAM

ATAM 活动

  • 业务/需求描述
  • 架构结构描述（视图、组件、连接）
  • 架构识别（关键场景、风险点、敏感点、权衡点）
  • 质量/效益树建立（质量属性和场景映射）
  • 架构方法分析（讨论现有方案、风险与权衡）
  • 场景优先级确认
  • 架构评估结论（风险建议与改进项）

SAAM 活动

  • 场景开发（收集代表性场景）
  • 架构描述/设计（架构视图表示系统）
  • 单一场景评估
  • 多场景交互评估
  • 总体评估（总结、风险点与改进点）

简言之：

  • SAAM：场景评估 + 支持度
  • ATAM：在此基础上加敏感/权衡、风险、优先级、改进建议

场景评估方法

  • 刺激（动词）
  • 刺激源（名词）
  • 响应
  • 响应度量
  • 制品（受到刺激后的一整套行为动作）
  • 环境

架构4+1 与 UML 4+1

  • 架构视图主要关注点是为什么要这么划分系统

  • UML 映射关系：

    • 逻辑视图 -> 包图、类图
    • 开发视图 -> 包图、组件图
    • 物理视图 -> 部署图
    • 进程视图 -> 活动图、序列图
    • 场景视图 -> 用例图、序列图

  • UML 视图：

    • 逻辑视图
    • 实现视图
    • 进程视图
    • 部署视图
    • 用例视图

逆向工程、正向工程、再工程

  • 逆向工程：通过分析程序，得到比源代码更高层次的程序表示过程，是一个程序的恢复过程
  • 正向工程：不仅获取程序设计信息，还对原程序进行一定重构
  • 再工程：逆向工程 + 新需求分析 + 正向工程
  • 设计恢复：通过工具获取程序的数据设计、过程设计和总体结构设计

ADL（架构描述语言）

三个基本元素：

  • 构件
  • 连接器
  • 架构配置

信息安全

五个特性：

  • 机密性
  • 安全性
  • 完整性
  • 可审查性
  • 可控性

区块链

特点：

  • 去中心化
  • 开放性
  • 自治性
  • 安全性
  • 匿名性

四大支柱：

  • 分布式
  • 智能合约
  • 密码学
  • 共识机制

视图的优点

  • 简化用户操作
  • 使业务逻辑独立
  • 保护机密信息
  • 提供新的数据库操作方式

分区和分表

相同点：

  • 增加查询速率
  • 减少 IO 压力
  • 使用分布式存储
  • 都针对数据表

不同点：

  • 分区本质上还是在一张表上
  • 分表是在不同的表上

反规范化

技术手段：

  • 增加派生列和冗余列
  • 重新组表、分割表

优点：

  • 查询快
  • 需要查询的表少
  • 连接操作少，统计快

缺点：

  • 冗余多
  • 修改同步更困难

备份

  • 热备份
  • 冷备份
  • 差量备份
  • 增量备份
  • 完全备份
  • 日志记录

设计模式

  • 创建型：工厂模式、单例模式
  • 结构型：适配器模式、代理模式
  • 行为型：观察者模式、模板模式

App 开发

两种模式：

  • 混合模式：利用 Web 技术运行在跨平台框架上，适合跨平台和快速迭代的项目，但硬件权限可能不足
  • 原生模式：直接调用系统 API，速度快
